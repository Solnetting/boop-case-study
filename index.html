The throughline

One question drove everything

"Can a stranger register their palm and pay with it, unassisted, in a real environment?"

Stakeholders came in with a feature-rich brief. Rather than build everything asked for, scope was cut to an MVP built to answer that single question. Everything else — vendor tooling, membership flows, the design system — was in service of proving or disproving it.

Story 1 · Discovery → Fix → Live validation

Pre-event test → one finding → shipped fix → 300 strangers

Every flow was paper-mapped before Figma. Low-fi went through ~9 iterations of the core user flow (V1–V9.2), three vendor-app versions, and separate bouncer/VIP/entrance variants — with friction risk tagged directly on the flow diagrams as they were built.

Before the event, a small-sample usability test with internal coworkers unrelated to the project surfaced a critical issue.

Finding Users didn't know which hand to start scanning with. Response Guidance screens: move closer, move further, align to centre, per-hand progress with haptic confirmation per scan.

The fix was designed directly from the finding: four states of haptic-and-visual guidance that let users calibrate their hand position without reading instructions. The phone becomes a coach, not a manual.

This was also when the device's own feedback language was designed: a light-ring system (colour + pulse) for charging, processing, success, and error states — entirely independent of the phone screen.

Purchases of goods, drinks, and food happened at physical kiosks via palm scan. The app was used afterward only to check balance and transaction history. The retail activity was the vehicle — the actual test was whether registration and payment worked, unassisted, for someone doing it for the first time.

Help-seeking was still fairly common — largely tied to hand-position confusion that the guidance screens reduced but didn't eliminate. Overall experience was rated positively despite that friction.

Story 2 · Stakeholder influence

Not just executing the brief

Stakeholders explicitly asked for a home screen dense with settings controls, designed for the live event demo. This is where design rationale overrode the brief.

What they asked for

What shipped

Situation Stakeholders explicitly asked for a home screen dense with settings controls for the event demo. My call Pushed back with design rationale — wallet-style home with balance and transaction history as the primary surface; settings to secondary/quick-access. Outcome Won the argument. The wallet-first home shipped. Add conversation specifics before presenting

The rationale: the event was a first-impression moment for real users, not a stakeholder demo. A settings-heavy home would have oriented the product around configuration rather than value — the opposite of what needed to be validated.

Secondary layer

A feedback language designed for the physical scanner itself — colour and pulse states independent of the phone screen. The device communicates state to both the user and the venue operator without requiring phone-glancing.

Colour Pulse State Meaning
Slow Charging Device is charging, not yet ready for use.
Steady Processing Scanning in progress — hold hand in position.
Single flash Success Palm recognised — payment or access approved.
Rapid Error Palm not recognised or transaction failed.
Story 3 · Technical trade-off

The decision that shaped V2

Two strong stories here — pick one before presenting. Both are available below.

The initial brief implied a World mini-app. A native React Native app was chosen explicitly to avoid install friction and dependency on the World app ecosystem.

Brief implied World mini-app — leverages the World ecosystem, no separate install needed. Decision Native React Native app — avoids install dependency and friction. Users shouldn't need to install World App to use Boop. Confirmed Explicitly documented in the "Requirements" meeting note. Aligned with Justin/Nathan as top-priority V2 decision, with a 6–7 week milestone runway to an April showcase.

Four competing UX concepts were evaluated for scanning guidance in V2. The core problem: the V1 guidance screens still required users to look at their phone instead of the scanner.

01
Static Guidance
Video tutorial before scanning. Requires users to watch and recall — adds cognitive load and another step before the actual gesture.
02
Dynamic Real-Time Feedback
Live coaching GIF on-screen during scanning. Solves real-time feedback but keeps the user's eyes on the phone, not the device.
03
Hybrid Mix
Lottie animation + progress bar. Richer feedback but still phone-centric — same fundamental attention-split problem.
04 →
Puck-First Confirm shipped
Minimal UI. Haptic and audio-led — the device does the coaching, the phone stays quiet. Haptics reserved for critical moments only. The happy path requires zero phone-glancing.

Reflection

What the loop taught

What worked
  • Pre-event test → fix → live validation loop proved the concept with real strangers, not proxies.
  • Scoping hard to one hypothesis made the event genuinely useful as a test, not just a demo.
  • Architecture decisions made early (one screen, 3–4 features, NFC tap) avoided mid-build pivots.
  • Pushing back on the settings-first home protected the product's identity at the highest-stakes moment.
What's still unresolved
  • Merchant/vendor dashboard — its own design problem, not yet started.
  • Accessibility audit — biometric-first flows carry exclusion risk; alternative auth paths are needed.
  • Wearable extension of the same gesture.
  • Hand-position confusion persisted even after the V1 guidance fix. Puck-First (V2) is the next attempt — not yet validated.

02 · The Problem

Contactless still needs a device.

Payment already went contactless, but it's still tied to a card or phone. Venues verifying age or membership means manual checks or extra hardware. The question: can the body become a universal key?

Stakeholders came in with a list of desired features. Scope was cut to an MVP because the goal wasn't to ship every idea — it was to validate one thing: can a stranger register their palm and pay with it, unassisted, in a real live environment?

03 · Research

What the category already proved

Amazon One / Tencent
Speed and 70%+ retention proven — but centralized data and regional silos limit trust and portability.
Redrock Biometrics / Arobix
Strong touchless accuracy — but no consumer ecosystem to convert that into daily behaviour.
World App / World ID
Decentralized identity via zero-knowledge proofs — the right privacy model, consumer UX still maturing.
Revolut / Banking apps
Financial hardware is already biometric-ready — proves the demand, but payments are still tied to a device.

04 · Process · V1

Paper first, then friction-tested

01
Every flow mapped on paper before any screen was drawn
Registration, payment, tickets, and the full app — all sketched first. Low-fi went through ~9 iterations of the user flow (V1–V9.2), 3 vendor-app versions, plus bouncer/VIP/entrance variants.
02
Flows were tagged live with friction markers during design
Annotating "potential friction" / "expected" directly on flows as they were designed — not after — kept the team honest before any testing happened.
03
Key architecture decisions made early
One main screen, 3–4 features max. NFC tap skips straight to pairing. No ID uploads in-app (bouncers check physical ID). Vendor side split into access control / bouncer / payment apps — phone-based, sharing a codebase.
Finding Pre-event usability test: users didn't know which hand to start with. Small sample, internal coworkers unrelated to the project. Response Guidance screens — move closer, move further, align to centre, per-hand progress, haptic confirmation per scan.
05 · Live Validation

~300 strangers, one real test

Purchases happened at the kiosk via palm scan — not inside the app. The app was for checking balance and history afterward. The retail transactions were the vehicle; registration and payment were what was actually being tested.

06 · Trade-offs

Every decision came with a rejected path

01
Hardware–mobile attention gap
Discovery Users watched their phone screen instead of the physical scanner during enrollment.
Decision Contextual haptic feedback — the phone "taps" a pattern when the user needs to adjust.
Rejected More on-screen instructional text — adds friction, doesn't fix the attention split.
02
Transaction legibility
Discovery Users couldn't recognize payments without a visual anchor to the venue.
Decision Establishment-specific iconography — Kiosk/Cafe badges for at-a-glance recognition.
Rejected Richer transaction detail pages — adds depth, not at-a-glance recognition.
03
Card-to-palm linking
Discovery Standard tap-to-link felt procedurally weak for something meant to feel permanent.
Decision Moving toward drag-and-drop / pull-to-link (future iteration, not yet shipped).
Rejected Shipping only the simpler tap flow long-term.
04
Card visual fidelity vs. API limits
Discovery Users wanted the digital card to match their physical card. API couldn't return card skins.
Decision "Universal Boop Card" aesthetic — branding and utility over plastic-mimicry. Sets digital-first expectations.
Rejected Engineering a workaround for skin data.
05
Settings-heavy home vs. wallet-first home
Discovery Stakeholders explicitly asked for a home screen dense with settings controls for the event demo.
Decision Pushed back — wallet-style home with balance/history as primary, settings as secondary quick-access.
Rejected Settings-forward IA as originally briefed. Add conversation specifics
06
Mini-app vs. native app (V2)
Discovery Initial ask implied a World mini-app.
Decision Native React Native app — avoids install dependency and friction.
Rejected Mini-app route — explicitly documented in the V2 requirements doc.
07
Scanning guidance — four proposals evaluated (V2)
Static Guidance — video tutorial
Dynamic Real-Time Feedback — live coaching GIF
Hybrid Mix — Lottie + progress bar
Puck-First — minimal UI, haptic/audio-led, no need to watch the screen
Decision Puck-First direction — haptics only on critical moments, happy path needs zero phone-glancing. Confirm shipped
Rejected The other three — all kept more attention on-screen, perpetuating the hardware–mobile attention gap.

08 · V2
Alpha — not yet live-tested

What came after

A native app with deeper World ID 4.0 integration, rethought registration guidance, and a multi-card wallet — everything the live test taught us, rebuilt from a stronger foundation.

01
Native React Native app
Avoids install dependency and friction. The mini-app route was explicitly dropped in the V2 requirements doc.
02
World ID 4.0 integration
Link an existing ID or create one in-flow — deeper but optional identity layer.
03
Multi-card wallet
Max 2 active cards. Last activated becomes primary — removes ambiguity about which card is charged.
04
Refined registration
Generic hand illustration with no handedness bias — mirrors whichever hand the user picks. Directly addresses the "which hand?" problem from V1.
09 · Results & Reflection

What the loop taught

What worked
  • Pre-event test → fix → live validation loop proved the concept with real strangers, not proxies.
  • The core concept clearly landed — people registered and paid with their palm in a real environment.
  • Architecture decisions made early (one screen, 3–4 features, NFC tap) avoided mid-build pivots.
  • Pushing back on the settings-first home protected the product's identity in the highest-stakes demo.
What's next
  • Merchant/vendor dashboard — its own design problem, not yet started.
  • Accessibility audit — biometric-first flows carry exclusion risk, alternative auth paths are needed.
  • Wearable extension of the same gesture.
  • Hand-position confusion persisted even after the V1 fix — Puck-First (V2) is the next attempt.
